## 2. Two cross-skill bugs to take back to paper-writer side

### 2a. paper_writer.sh 0.6.3 calls the pre-v0.6.0 adversarial CLI shape

Your Claude session shows the orchestrator invoking:

```
beril-adversarial --type paper <project_id>
```

The current adversarial CLI (v0.6.0+) expects:

```
beril-adversarial review --type paper <draft_dir>
```

Two breaking changes happened at v0.6.0:
- `review` subcommand became required (CLI gained subcommand structure).
- `<draft_dir>` is now positional, not `<project_id>`. The reviewer reads from a per-draft directory layout (paper-writer v0.6+ shape: `papers/draft_N/draft.md`, `papers/draft_N/citation_pool.json`, etc.), not from the BERIL project root.

The producer (us) followed CONTRACT.md from v0.6.0 forward. The consumer (paper-writer) didn't update integration code to use the new CLI surface. **CONTRACT.md alone is not preventive** — it's a target the consumer has to read and update against.

### 2b. paper_writer.sh silently accepts CLI usage error AS THE REVIEW

The compounding bug: when paper-writer invoked the wrong CLI shape, argparse emitted a 10-line usage error to stderr, exit code 2. paper-writer's orchestrator captured stderr, called it `audit/adversarial_review.md`, and shipped it as the canonical adversarial review for draft_9. No defensive check that the file is non-trivial.

**Recommended fix on paper-writer side:** a smoke test in paper-writer's repo that runs the full pipeline against a fixture project and asserts the adversarial review file:
- Exit code 0 from the CLI invocation.
- File exists and is >50 lines.
- Contains expected `## P0` or `## Critical` headers (indicating real findings or a real PASS, not a CLI error).
- The `.json` companion parses as valid JSON and contains a `findings` array.

That smoke test is consumer-side responsibility — the producer (us) can't enforce that you call us correctly. Our `tests/integration/test_paper_writer_interop.py` exercises the adversarial orchestrator with synthetic paper-writer-shaped inputs; it does NOT invoke paper_writer.sh's actual orchestrator. So the producer-side test passes while the consumer-side bug ships.

**Producer-side help we'll add at next release** (cheap, suggested by your incident): when a v0.5.x-shape invocation hits the v0.6+ CLI, the argparse usage error message will include a hint like `"This CLI changed in v0.6.0 — see CONTRACT.md for migration."` Doesn't fix the bug, but makes the next consumer's debugging cycle faster.

### missing_section decomposition observation

Your team's note about whether `missing_section` should split into `rendering_failure` / `methods_gap` / `compliance_gap` is logged for v0.7.0 consideration. We're already planning two schema changes for v0.7.0:
1. Rename `narrative_weakness` → `central_objection` (label issue, not function issue — flagged by your taxonomy review).
2. Add `citation_reality` to the presentation schema (currently paper-only; presentation-maker team asked for parity).

Decomposing `missing_section` would be a third v0.7.0 schema break. Initial read: **maybe not worth the schema complexity**. The three sub-classes you'd carve out have very different remediation paths (rendering = pipeline bug, methods = author rewrite, compliance = ICMJE-level checklist) but the reviewer's job is the same in all three: tell the author "this is missing/broken, here's where." A `subkind` enum on `missing_section` rather than three separate classes might capture the distinction without breaking downstream consumers' classification code.

We'd want one more data point — does the lack of subkind genuinely make your remediation harder, or is the per-finding recommendation text already steering you correctly? If it's the former, we'll add it as a third v0.7.0 schema change. If it's the latter, we'll skip it.
